# SPEC.md

## 仓库用途

本仓库用于整理不同任务场景下的 AI 工作流和 Prompt 模板，包括论文阅读、领域论文检索、科研代码生成、科研展示生成、通用 Prompt 结构等内容。

核心目标是让 AI 辅助工作更可复用、更可检查。每个工作流都应清楚说明适用场景、所需输入、执行流程、输出要求和约束条件。

## 目录说明

- `templates/`：通用模板，包括工作流模板、Prompt 模板和输出格式模板。
- `paper-reading/`：用于论文阅读、理解和分析的 Prompt 工作流。
- `domain-paper-search/`：用于特定领域论文检索、筛选和整理的 Prompt 工作流。
- `code-generation/`：用于科研代码生成、理解和修改的 Prompt 工作流。
- `presentation-generation/`：用于将论文、笔记、项目材料等科研内容转化为 PPT 逐页规划和 PPTX 生成规格的 Prompt 工作流。

其中 `templates/interactive-controller-template.md` 是通用交互控制模块，用于定义 Ask-First / Interactive Mode 的需求澄清门控。面向 ChatGPT 网页版日常使用时，更推荐将该控制模块融合进专业任务模板，形成完整的 `*-interactive.md` 文件。

## 共享文档约定

- `AGENTS.md` 定义 Codex 或其他代码助手在本仓库中的工作方式。
- `SPEC.md` 定义仓库的共同目标、目录结构和文档规范。
- 这两个文件应提交到 Git，以便在不同设备上保持一致的 Codex 开发习惯。

## 本地任务记录

- 使用 `local-tasks/task001.md`、`local-tasks/task002.md` 等文件记录具体任务。
- 本地任务文件应包含任务背景、任务目的、任务执行方法、任务步骤和验收标准。
- `local-tasks/` 已通过 `.gitignore` 忽略，因为这些文件属于个人工作记录，不属于共享仓库文档。

## Prompt 模板规范

新增或更新工作流 Prompt 时，优先采用以下结构：

1. 元数据或标题
2. 使用场景
3. 所需输入
4. 角色设定
5. 任务目标
6. 执行流程
7. 输出要求
8. 约束条件
9. 失败处理
10. 可复用变量或示例用法

具体标题可以根据任务调整，但工作流应保持易复用、易改写、易执行。

对于重要任务 Prompt，默认同时维护两种版本：

- Direct Mode：通用执行版本，适合需求已经明确时直接执行。
- Interactive Mode / Ask-First Mode：交互询问版本，文件名通常使用 `*-interactive.md`，适合需求尚不完全明确时先提问、再执行。

交互版模板不替代直接执行版模板。新增交互版时，应尽量复用对应 Direct Mode 的专业任务逻辑，只增加需求澄清门控。日常使用时，不建议让用户每次上传“总控模板 + 专业模板 + 文件”，更推荐上传“融合好的交互版专业模板 + 文件”。

## 内容准则

- 指令应足够具体，让另一个 AI 助手不需要猜测也能执行。
- 当工作流涉及分析任务时，应区分已确认事实、合理推断和未知信息。
- 输出格式应可操作，避免只有宽泛总结。
- 通用模板中不要加入缺乏依据的事实、结论或领域判断。
- 不同任务族的 Prompt 应放在不同文件或目录中，避免混杂。

## 跨设备 Codex 使用

本仓库应将可共享的 Codex 使用规则保存在可提交的 Markdown 文件中，同时将每个任务的临时记录保留在本地。这样可以在不同设备之间同步稳定的工作习惯，又不会把个人任务计划和临时上下文提交到仓库中。
