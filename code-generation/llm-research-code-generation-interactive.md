# LLM 科研实验代码生成与修改交互执行模板

## 0. 使用场景

本模板适合在 ChatGPT 网页版中上传科研代码仓库、关键源码、配置、日志或需求说明后使用。它会先澄清任务类型、修改边界、输出形式和验证方式，再正式进行代码理解、修改方案设计或代码生成。

适合以下场景：

- 用户还不确定是要理解代码、修复 bug、添加功能、重构仓库，还是生成新模块。
- 用户希望先确认哪些文件、接口、配置或实验协议不能改。
- 用户希望 ChatGPT 不要在需求不清时直接生成代码或 patch。

如果需求、约束和输出形式已经完全明确，可以直接使用 `code-generation/llm-research-code-generation-prompt.md`。

## 1. 交互式执行规则

你需要按两个阶段工作：

1. 阶段一：需求澄清阶段；
2. 阶段二：正式执行阶段。

在用户明确说出以下任一类确认指令之前，你只能进行代码理解、需求澄清和方案边界确认，不能输出最终代码、完整 patch/diff、完整替换文件或最终修改结果：

- “开始实现”
- “可以开始”
- “按当前信息执行”
- “不用再问了”
- “直接开始”
- 其他语义等价的明确确认

每轮最多提出 5 个问题。只问会影响代码修改正确性、实验可复现性或输出可用性的问题。

## 2. 阶段一：需求澄清

在澄清阶段，请先阅读用户上传的代码仓库、目录结构、核心文件、配置、日志、报错信息和任务说明。

你必须按以下结构输出：

```md
## 我对任务的初步理解

## 当前还需要确认的问题

## 默认假设

## 是否可以开始
```

如果信息已经足够，应提示：

> 当前信息已经基本足够。如果你认可上述理解和默认假设，可以回复“开始实现”，我将进入正式代码分析与修改阶段。

如果用户不回答部分问题，可以使用默认假设继续，但必须在正式执行前列明这些假设。

## 3. 本任务专属澄清问题

优先从以下问题中选择最关键的问题提问，每轮不超过 5 个：

1. 本次任务类型是什么：理解代码、修复 bug、添加功能、重构仓库、生成新模块，还是复现实验问题排查？
2. 是否允许修改目录结构、新增文件、调整配置字段或改变命令行参数？
3. 期望输出是什么：完整文件、patch/diff、关键代码片段、修改方案，还是只给排查思路？
4. 哪些文件、接口、函数签名、配置、数据处理流程、评价指标、baseline 或实验协议不能改？
5. 是否需要给出可直接运行的验证命令、最小测试或训练 / 评估 / 推理命令？
6. 当前运行环境是什么，例如 Python、CUDA、PyTorch、Transformers 或其他关键依赖版本？
7. 是否存在必须兼容的已有 checkpoint、日志格式、输出目录或结果文件？
8. 是否允许指出但暂不修改与本次目标无关的潜在问题？

## 4. 阶段二：正式执行

只有用户明确确认开始后，才进入正式执行阶段。

正式执行时，请按照 `llm-research-code-generation-prompt.md` 的科研代码生成与修改逻辑工作。至少应覆盖：

1. 完整阅读并理解用户提供的代码、配置、日志和说明。
2. 复述项目目标、关键入口、核心流程、配置与脚本关系，以及当前问题可能位置。
3. 判断当前问题的直接原因、可能原因、不确定信息和需要用户确认的信息。
4. 先给出修改方案，再给出代码。
5. 说明修改哪些文件或模块、每处修改解决什么问题、为什么选择该方案、是否影响兼容性以及如何验证。
6. 生成代码修改时保持局部、清晰、可回滚，不添加未要求功能，不重构无关代码。
7. 对科研实验相关逻辑保持谨慎，不擅自改变数据划分、预处理、评价指标、随机种子、baseline 或核心实验设定。
8. 说明每处修改的位置、意义和影响。
9. 给出可执行的验证方法，包括环境检查、运行命令、最小测试、预期输出和失败排查方向。
10. 最后进行质量自查，确认输出可以直接用于实际修改代码。

如果当前信息不足以生成完整代码或验证命令，必须明确说明缺少什么，并给出基于已有文件的合理建议。

## 5. 输出要求

正式输出使用中文，代码、路径、命令、函数名和配置字段保持原文格式。

最终输出应遵循以下结构：

```md
# 代码修改结果

## 1. 原代码理解

## 2. 问题判断

## 3. 修改方案

## 4. 具体代码修改

## 5. 修改清单

## 6. 完整文件集合

## 7. 运行与验证

## 8. 风险与后续建议
```

如果用户只要求理解代码或只要求修改方案，可以省略不适用的小节，但必须说明原因。

在正式执行前，不得输出最终代码、完整 patch/diff 或最终替换文件；在正式执行后，不得因为交互模式而削弱原代码模板对最小修改、可验证性和科研实验可靠性的要求。
