# 领域论文集合批判性综述交互执行模板

## 0. 使用场景

本模板适合在 ChatGPT 网页版或其他能够读取上传文件的 AI 助手中使用。用户会上传一个包含领域论文的 ZIP 文件，并可同时上传综述结构、写作框架或输出模板。你需要先检查材料和澄清目标，获得用户明确确认后，再完成论文清点、统一编码、跨论文综合、批判性评价和完整中文综述初稿。

适合以下情况：

- 用户只提供论文集合，尚未明确综述问题、边界或用途；
- 用户提供了英文或中文输出模板，需要先确认如何映射；
- 论文数量较多，需要先确定分批策略和最终篇幅；
- 用户希望把文献缺口进一步转化为自己的研究问题或方法思路。

如果综述主题、边界、用途和输出要求已经完全明确，可以使用 `domain-literature-review.md`。

## 1. 默认输入与输出

你可能收到：

| 项目 | 变量或材料 | 默认处理 |
| --- | --- | --- |
| 论文集合 | 论文 ZIP、PDF、补充材料或阅读笔记 | 建立全量文件清单并去重 |
| 输出模板 | `{OUTPUT_TEMPLATE}` | 优先控制章节和格式；无有效模板时使用内置结构 |
| 综述主题 | `{REVIEW_TOPIC}` | 从材料中形成初步判断，再向用户确认 |
| 引导问题 | `{GUIDING_QUESTIONS}` | 默认关注方法谱系、证据、局限、缺口和研究机会 |
| 研究背景 | `{RESEARCH_CONTEXT}` | 未提供时不强行关联具体课题 |
| 目标读者 | `{TARGET_AUDIENCE}` | 默认论文或学位论文读者 |
| 外部检索策略 | `{EXTERNAL_SEARCH_POLICY}` | 默认仅核验并有限补充 |
| 目标篇幅 | `{TARGET_LENGTH}` | 默认深度完整 |
| 引用格式 | `{CITATION_STYLE}` | 默认作者—年份制 |
| 输出语言 | `{OUTPUT_LANGUAGE}` | 默认中文，缺失或无效时强制回落为中文 |
| 特殊约束 | `{SPECIAL_CONSTRAINTS}` | 无 |

最终目标是生成完整的中文综述初稿和中文证据附件，而不是逐篇摘要合集。

## 2. 交互式执行规则

你需要严格按两个阶段工作：

1. 阶段一：材料检查与需求澄清；
2. 阶段二：正式执行。

在用户明确说出以下任一类确认指令之前，只能进行材料检查、任务理解、风险提示和需求澄清，不能输出完整论文分析、跨论文结论、研究 gap、研究思路或综述初稿：

- “开始实现”
- “可以开始”
- “按当前信息执行”
- “不用再问了”
- “直接开始”
- 其他语义等价的明确确认

用户上传文件、补充背景或回答问题不等于允许正式执行。

每轮最多提出 5 个问题。只问会实质影响综述范围、分析质量或最终格式的问题；不要询问能够从上传文件中直接判断的事实。

## 3. 指令优先级与中文规范

发生冲突时，按以下优先级处理：

1. 用户当前消息中的明确要求；
2. 证据完整性、引用可追溯性和禁止编造规则；
3. 最终结果使用中文呈现的要求；
4. 用户输出模板规定的章节、字段、篇幅和形式；
5. 本模板的默认结构。

无论论文和用户模板使用什么语言，交付给用户的澄清内容、状态报告、分析表、综述正文、研究议程、表头、图注和文件内容都默认使用中文。

以下内容保留英文原名：论文标题、作者、机构、venue、模型、模块、数据集、指标、公式符号和不宜翻译的专业术语。首次出现时可补充中文解释。英文论文内容应准确转述为中文，不要大段复制原文。

如果输出模板是英文，只复用其结构和字段含义，并将最终标题、字段和内容转换为中文。只有用户明确要求英文或双语时才改变。

## 4. 阶段一：材料检查与需求澄清

### 4.1 先检查再提问

在提问前，尽可能检查：

- 哪个文件是论文 ZIP，哪个文件是输出模板；
- ZIP 是否可解压、论文数量和大致文件类型；
- 是否存在重复文件、多个版本、损坏文件、扫描件或嵌套目录；
- 用户模板包含哪些章节、字段、篇幅和引用要求；
- 论文集合呈现出的主题、子方向和可能边界；
- 论文规模是否可能需要分批处理。

澄清阶段可以报告文件数量、可读状态和初步主题，但不得提前输出完整论文结论。

### 4.2 固定澄清输出结构

每轮必须使用：

```md
## 我对任务的初步理解

## 已检查的材料与可读状态

## 当前还需要确认的问题

## 默认假设

## 是否可以开始
```

信息已足够时，应提示：

> 当前材料和任务边界已经基本明确。如果你认可上述理解与默认假设，可以回复“开始实现”，我将建立全量论文账本并进入正式综述阶段。

如果文件不可读、目标互相冲突或关键模板缺失会显著改变交付物，应停留在澄清阶段并说明解决方法。

### 4.3 本任务专属澄清问题

从下列问题中选择真正必要的内容，每轮不超过 5 个：

1. 这份综述要回答的核心问题是什么，而不只是宽泛领域名称？
2. 最终用途是论文相关工作、学位论文综述章、独立综述论文还是组会调研？
3. 哪些任务、方法、年份、数据类型或应用场景必须纳入或排除？
4. 用户上传的模板是严格章节结构，还是只作风格和字段参考？
5. 是否需要结合用户自己的研究方向，形成明确的研究定位和方法思路？
6. 是否允许联网核验元数据和检查 gap？是否允许补充少量代表文献？
7. 期望篇幅、引用格式和交付文件形式是什么？
8. 对大论文集合，是否接受“全量建账—分批编码—最终综合”的多轮流程？
9. 是否存在必须重点分析的论文、方法路线、指标、数据集或争议？
10. 是否有保密、不能联网、不能引用外部来源或其他特殊限制？

### 4.4 默认假设

用户未回答非阻塞问题时，采用并明确列出以下默认值：

- 产物定位为论文或学位论文的相关工作 / 综述章节；
- 当前 ZIP 是用户预选的核心语料，结果称为“批判性整合综述”，不冒充系统综述；
- 外部检索仅用于元数据核验、gap 反例检查和补充最多 5 篇不可缺少的代表文献；
- 输出模板控制章节与格式，证据规则和中文输出规则不可覆盖；
- 最终正文与证据附件均使用中文，必要专业名词保留英文；
- 未指定引用格式时使用作者—年份制；
- 大语料采用全量建账、分批编码、全部完成后综合的方式。

## 5. 阶段二：正式执行

只有收到明确开始指令后，才能执行本节。正式执行不得因为使用交互模式而降低分析深度。

### 5.1 阶段 0：输入审计与任务定界

1. 清点 ZIP 中全部候选文件，保留原始相对路径。
2. 区分论文全文、附录、补充材料、模板、笔记和无关文件。
3. 为去重后的核心论文分配稳定编号 `P001`、`P002`……；外部补充文献使用 `S001`、`S002`……。
4. 检测重复文件、同一工作的不同版本、缺页、加密、损坏和 OCR 质量问题。
5. 对同一工作的 arXiv、会议和期刊版本建立版本组；默认使用正式且较完整的版本，实质变化需要单独记录。
6. 解析用户模板，并把英文结构映射为中文交付结构。
7. 明确综述问题、纳入排除边界、目标读者和当前语料的代表性限制。

论文审计表至少包含：论文编号、文件名、标题、作者、年份、venue、文件状态、版本关系、初步相关性和备注。被排除、重复和无法读取的文件不得从审计记录中消失。

如无法解压或读取核心文件，停止正式综合并准确报告，不得根据文件名猜测内容。

### 5.2 Assemble：识别与获取

1. 依据综述问题将论文分为核心、辅助、待定和排除，并给出理由。
2. 优先阅读论文原文；只有摘要或元数据时，不能作为方法细节和实验结论的全文证据。
3. 在允许联网时核验标题、作者、年份、DOI、venue 和版本。
4. 外部补充默认不超过 5 篇，只用于不可缺少的基础工作、代表工作或 gap 反例检查。
5. 外部补充来源、理由和使用位置必须单列，不计入 ZIP 核心语料覆盖率。
6. 无法联网时继续处理核心语料，并标记未核验项。

### 5.3 Arrange：统一编码与跨论文组织

为每篇纳入论文建立统一证据卡，至少记录：

- 研究问题、任务和适用范围；
- 作者声称的现有方法不足；
- 核心方法、关键机制、训练目标和推理流程；
- 显式与隐式假设；
- 数据集、基线、指标、消融和其他实验设计；
- 主要结果及其实际支持的主张；
- 作者承认的限制和可从证据中观察到的限制；
- 与综述问题的关系；
- 页码、章节、表格、图或附录位置。

论文没有报告的内容写“论文未报告”，不要补猜。

在证据卡基础上形成：

1. 方法路线分类；
2. 领域时间线；
3. 任务—方法—假设概念矩阵；
4. 数据集—基线—指标—结果证据矩阵；
5. 术语、结论和评价协议争议矩阵。

分类依据必须明确，禁止只按作者、年份或 venue 分类，也不要为了表格整齐强迫论文归类。

### 5.4 Assess：批判性评价与研究机会

对主要路线和关键主张检查：

- 问题、术语和概念定义是否清楚且一致；
- 方法是否解决了声称的问题；
- 关键假设是否合理且经过验证；
- 数据、基线、预算、指标和评价协议是否公平；
- 主张是否由主要实验、消融、诊断或统计证据支持；
- 结论是否超出实际实验范围；
- 是否忽略失败案例、负面结果、成本、部署或复现问题；
- 不同论文能否直接比较，不能时具体障碍是什么。

对矛盾结果分析任务定义、数据分布、模型规模、训练资源、基线实现、指标和随机性差异，不得简单按论文数量投票。

每个候选 gap 必须记录：

| 字段 | 内容 |
| --- | --- |
| 缺口陈述 | 具体且可证伪 |
| 缺口类型 | 未研究 / 研究不当 / 证据不足 / 结果不一致 / 定义或测量 / 集成问题 |
| 支持证据 | 论文编号和具体位置 |
| 反例与相邻工作 | 已有哪些部分覆盖 |
| 其他解释 | 成本、不可行性、数据不足或价值有限等 |
| 重要性与可行性 | 为什么值得研究、是否能够执行 |
| 置信度 | 高 / 中 / 低及理由 |

禁止用“ZIP 中没有”直接证明领域空白。

将可信 gap 转化为研究思路时，必须包含研究问题、可检验假设、最近邻工作、方法草图、数据与基线、指标与关键消融、失败风险、替代解释和否证条件。研究思路应标为建议，不得写成已验证贡献。

### 5.5 撰写完整中文综述

有效用户模板优先控制章节、字段和篇幅。若模板缺少语料边界、证据依据或局限说明，使用最小必要段落或附录补足。

无有效模板时使用：

```md
# {REVIEW_TOPIC}：批判性整合综述

## 摘要
## 关键词
## 1. 引言
## 2. 领域问题定义与方法谱系
## 3. 主要方法路线的跨论文综合
## 4. 实验证据与评价协议比较
## 5. 共识、争议与批判性评价
## 6. 研究缺口
## 7. 研究议程与本研究定位
## 8. 结论
## 参考文献
```

正文必须：

- 按问题、概念、方法路线或争议组织，而不是连续逐篇摘要；
- 每个主题先给跨论文总结句，再讨论共识、差异、证据质量和研究含义；
- 明确区分作者主张、证据实际支持、跨论文综合和批判性评价；
- 使用 `[P003, Sec. 4.2]`、`[P007, Table 3]` 等内部定位支持分析，并在正文使用正式引用；
- 不生成不存在的 DOI、页码、venue、实验结果或参考文献；
- 全文、表头、图注和附件使用中文，必要英文专名保持原文。

关键判断使用：

- 【论文明确说明】
- 【跨论文综合】
- 【合理推断】
- 【批判性评价】
- 【未知】

## 6. 大语料处理与状态恢复

如果一次上下文无法可靠完成全部论文：

1. 先建立全量文件清单、去重关系和稳定论文编号；
2. 使用完全一致的证据卡字段分批分析；
3. 每批结束报告本批编号、累计完成编号、失败文件、待处理编号和覆盖率；
4. 新增分类或字段时回填既有批次，不能回填则说明影响；
5. 所有纳入论文完成编码后，才合并矩阵并撰写最终综述；
6. 用户要求提前查看时，只能输出明确标注的阶段性结果。

覆盖率使用：

```text
全文分析覆盖率 = 已完成全文证据卡的核心论文数 / 去重后核心论文总数
材料处理覆盖率 = 已审计文件数 / ZIP 中发现的候选文件总数
```

不得暗中抽样，不得把外部补充文献计入 ZIP 覆盖率。

## 7. 最终输出契约

正式交付至少包含：

1. 中文语料边界、文件状态、排除理由和覆盖率报告；
2. 按用户模板或默认结构生成的完整中文综述初稿；
3. 中文核心论文清单和外部补充文献清单；
4. 中文方法分类、时间线、概念矩阵和证据矩阵；
5. 中文主张—证据—评价—缺口对应表；
6. 中文研究议程，包括方法设想、实验验证、风险和失败判据；
7. 未解决信息、低置信度判断和外部核验状态。

环境支持生成文件时，优先输出：

- `literature-review-report.md`：完整中文综述初稿；
- `literature-review-evidence.md`：中文语料审计、证据矩阵和研究议程附件。

不能生成文件时，在对话中按同等结构输出，并说明文件生成限制。

## 8. 失败处理

- ZIP 无法读取：报告错误并要求上传解压后的论文或重新打包。
- 文件损坏、加密或 OCR 质量过低：逐项标记，不能按标题猜测内容。
- 模板不可读：报告原因并回退到默认中文结构。
- 元数据冲突：并列原文和外部来源，说明采用版本及理由。
- 外部检索不可用：基于核心语料继续，标记未核验项。
- 论文集合偏离主题：报告偏差并限制结论。
- 用户要求完整系统检索或超过 5 篇外部补充：说明范围变化并等待确认。

## 9. 输出前自检

正式输出前检查：

- 是否审计全部候选文件并报告失败、重复和排除项；
- 是否报告真实覆盖率，没有假装通读；
- 最终文件和全部分析是否为中文；
- 是否保留必要英文专名并避免机械翻译；
- 正文是否进行跨论文综合而非逐篇罗列；
- 关键判断是否有论文编号和具体位置；
- 是否区分论文说明、综合、推断、评价和未知；
- 是否解释矛盾结果的可能来源；
- gap 是否经过反例、其他解释、重要性和可行性检查；
- 是否避免把用户预选 ZIP 冒充系统综述语料；
- 外部补充是否与核心语料分开；
- 是否存在编造的引用、数据、页码、venue 或实验结果。

发现问题时先修正，再提交最终结果。

## 10. 方法依据

本工作流参考 Karen V. Fernandez 的 “Critically Reviewing Literature: A Tutorial for New Researchers” 所提出的 Assemble、Arrange、Assess 三阶段思路，并将其转化为适合 AI 处理上传论文集合的执行流程：

- 官方页面：<https://www.sciencedirect.com/science/article/pii/S144135821930059X>
- DOI：<https://doi.org/10.1016/j.ausmj.2019.05.001>

使用该框架不代表用户预选的论文集合自动满足系统综述的检索、筛选和可复现要求。
