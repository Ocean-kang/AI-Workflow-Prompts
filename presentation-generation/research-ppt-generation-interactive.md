# 科研 PPT 生成交互执行模板

## 0. 使用场景

本模板适合在 ChatGPT 网页版中上传 PDF、Markdown、Word、文本笔记或少量辅助表格后使用，用于先澄清展示目标、受众、页数、深度和设计限制，再生成可用于制作 PPTX 的逐页结构化规格。

适合以下场景：

- 用户知道要做科研 PPT，但还没有明确展示时长、受众或页数。
- 输入材料较长，需要先判断哪些内容适合放进 PPT。
- 用户希望先确认默认假设，再让 ChatGPT 输出正式的 PPT 制作蓝图。
- 用户希望最终结果仍保持 `research-ppt-generation.md` 的完整结构和质量要求。

如果需求已经完全明确，可以直接使用 `presentation-generation/research-ppt-generation.md`。

## 1. 交互式执行规则

你需要按两个阶段工作：

1. 阶段一：需求澄清阶段；
2. 阶段二：正式执行阶段。

在用户明确说出以下任一类确认指令之前，你只能进行需求澄清，不能输出完整 PPT 规划、完整逐页方案或完整 PPTX 生成规格：

- “开始实现”
- “可以开始”
- “按当前信息执行”
- “不用再问了”
- “直接开始”
- 其他语义等价的明确确认

如果用户只是上传文件、补充背景、回答问题或描述展示想法，不代表已经允许你进入正式执行阶段。

每轮最多提出 5 个问题。只问会显著影响 PPT 结构、信息取舍或展示效果的问题。不要询问可以从上传文件或用户说明中直接判断的信息。

## 2. 阶段一：需求澄清

在澄清阶段，请先阅读用户上传的文件、任务说明和已有上下文，然后形成初步理解。

你必须按以下结构输出：

```md
## 我对任务的初步理解

## 当前还需要确认的问题

## 默认假设

## 是否可以开始
```

如果信息已经基本足够，应提示：

> 当前信息已经基本足够。如果你认可上述理解和默认假设，可以回复“开始实现”，我将进入正式科研 PPT 生成阶段。

如果用户不回答部分问题，可以基于默认假设继续，但需要在正式执行前说明这些假设。

## 3. 本任务专属澄清问题

优先从以下问题中选择最关键的问题提问，每轮不超过 5 个：

1. 这份 PPT 的展示场景是什么：论文汇报、组会分享、课题进展、项目总结、课程展示、答辩预演，还是其他？
2. 目标受众是谁：同方向研究者、跨方向同学、导师/课题组、课程老师、评审专家，还是非专业听众？
3. 展示时长和目标页数是多少？如果不确定，是否采用默认 10–15 分钟、10–15 页？
4. 技术深度希望是什么：快速概览、标准组会深度、深入方法与实验、答辩级细节？
5. 是否需要讲者备注或口播稿？
6. 是否有必须包含的内容，例如方法图、实验结果、消融、局限性、未来工作、个人理解或与自己课题的关系？
7. 是否有必须避免的内容，例如公式太多、实验细节太多、背景太长、过度评价、商业风表达？
8. 是否已有 PPT 模板、配色、学校/课题组风格或页面比例要求？
9. 是否希望输出更偏人工制作蓝图，还是更偏结构化 JSON/PPTX 生成指令？
10. 是否需要把某些图片、表格或公式指定到特定页面？

如果用户没有明确回答，默认采用：

- 展示场景：通用科研展示；
- 受众：有科研背景但不一定熟悉该具体方向的听众；
- 时长：10–15 分钟；
- 页数：10–15 页；
- 技术深度：科研组会标准深度；
- 输出：Markdown 逐页规划 + JSON PPTX 生成规格；
- 风格：简洁、专业、克制。

## 4. 阶段二：正式执行

只有用户明确确认开始后，才进入正式执行阶段。

正式执行时，请复用 `research-ppt-generation.md` 的任务逻辑。至少应覆盖：

1. PPT 总体设计说明：标题、展示目的、目标受众、建议页数、建议时长、整体叙事线。
2. 内容提炼：核心结论、关键贡献/发现、必须呈现的证据、建议省略或放入备份页的内容。
3. 逐页 PPT 规划表：页码、页面标题、页面目标、核心信息、正文 bullet、建议视觉元素、版式建议、讲者备注、信息来源。
4. 图表与视觉建议：说明哪些内容适合做流程图、对比表、实验结果图、时间线、架构图或总结页。
5. PPTX 生成规格：使用结构化 JSON 描述 slide list、layout、title、bullets、visual、speaker_notes 和 style_constraints。
6. 质量自检清单：检查是否简洁、是否有证据来源、是否避免堆字、是否适合目标受众、是否存在无法确认的信息。

正式输出应服务于用户确认的展示用途。如果用户提供了模板、页面比例、配色或内容优先级，以用户最新确认的信息为准。

## 5. 输出要求

需求澄清阶段的输出应简洁、可回答、可决策。不要输出完整 PPT 规划表或完整 JSON 规格。

正式执行阶段请按照以下结构输出：

```md
# {PRESENTATION_TITLE} 科研 PPT 生成方案

## 1. PPT 总体设计说明

## 2. 内容提炼

## 3. 逐页 PPT 规划表

## 4. 图表与视觉建议

## 5. PPTX 生成规格

## 6. 质量自检清单
```

正式输出默认使用中文，关键英文术语保留原文。

## 7. 质量与事实约束

正式执行时必须遵守：

1. 不编造上传材料中没有的结论、实验结果、数据数值、作者观点或项目进展。
2. 缺失信息必须标注“材料中未明确说明”。
3. 关键判断尽量区分：
   - 【材料明确说明】
   - 【合理推断】
   - 【建议性改写】
4. 每页正文 bullet 默认 3–5 条，每条尽量短。
5. 输出目标是帮助生成简洁体面的科研 PPT，避免营销风、花哨风和大段文字堆叠。
6. 如果材料包含 Excel / CSV / 表格数据，只作为辅助证据提炼；除非用户明确要求，不展开完整数据分析报告。
7. 如果无法确认某页图表所需数据，请在图表建议和 JSON 规格中明确标注。

## 8. 失败处理

如果上传文件无法读取、信息缺失或任务目标互相冲突，应停留在需求澄清阶段，并说明：

- 哪些信息已经确认；
- 哪些信息缺失或冲突；
- 缺失信息会影响什么；
- 如果用户不补充信息，将采用什么默认假设。

如果用户要求跳过澄清并直接执行，可以进入正式执行阶段，但必须先列出关键默认假设。
